# 汉字消消乐开发任务拆解文档

## 1. 文档信息
- 文档名称：汉字消消乐开发任务拆解
- 适用范围：仅任务规划，不包含编码实现
- 对应文档：拼音消消乐需求.md、拼音消消乐技术文档.md
- 文档日期：2026-03-29

## 2. 任务拆解原则
- 以“可交付、可验收、可回归”为最小粒度。
- 任务按依赖顺序编排，避免并行冲突。
- 每项任务必须有 DoD（Definition of Done）。

## 3. 总体阶段
1. 阶段A：工程与基础设施
2. 阶段B：核心业务能力
3. 阶段C：质量与稳定性
4. 阶段D：发布与验收

## 4. 详细任务清单

### A. 工程与基础设施

#### A1. 项目脚手架与规范
- 目标：建立可持续迭代的前端工程。
- 子任务：
  - 初始化 Vue3 + TS + Vite 项目。
  - 配置 ESLint、Prettier、EditorConfig。
  - 建立目录规范（pages、components、domain、data、store、utils）。
- DoD：
  - 本地可启动、可构建。
  - 代码风格检查可通过。

#### A2. 路由骨架与页面占位
- 目标：打通页面跳转链路。
- 子任务：
  - 配置五个核心路由。
  - 完成页面空骨架与返回链路。
  - 路由参数基础校验与回退。
- DoD：
  - 首页 -> 单元 -> 关卡 -> 游戏 -> 通关 跳转可达。
  - 非法参数不白屏。

#### A3. 词库数据层
- 目标：建立教材数据读取能力。
- 子任务：
  - 定义词库模型。
  - 实现按册别/单元读取接口。
  - 增加基础数据校验（缺失、冲突）。
- DoD：
  - 可稳定输出指定单元词条。
  - 数据异常可提示并兜底。

### B. 核心业务能力

#### B1. 关卡构建引擎
- 目标：从词库生成可玩关卡。
- 子任务：
  - 实现关卡切片（每关最多 12 词）。
  - 实现复习补词逻辑。
  - 实现卡片生成与洗牌。
- DoD：
  - 每关稳定输出 24 张卡片。
  - 复习补词逻辑符合需求。

#### B2. 游戏交互引擎
- 目标：实现完整配对玩法。
- 子任务：
  - 两张卡片选择与状态控制。
  - 匹配判定、错误回退、计错。
  - 通关判定与延时跳转。
- DoD：
  - 正确/错误配对行为符合需求文档。
  - 连点不引起状态错乱。

#### B3. 计时、评分与统计
- 目标：实现可追踪成绩体系。
- 子任务：
  - 实现计时器生命周期管理。
  - 星级计算函数。
  - 最佳成绩覆盖逻辑。
- DoD：
  - 通关页可显示正确星级与耗时。
  - 同关重复游玩时最佳成绩更新正确。

#### B4. 进度与解锁系统
- 目标：实现可持续学习链路。
- 子任务：
  - 本地进度读写封装。
  - 关卡解锁与单元解锁规则实现。
  - 版本迁移逻辑实现。
- DoD：
  - 刷新后进度保留。
  - 跨关卡/跨单元解锁正确。

#### B5. 提示与声音模块
- 目标：增强学习反馈体验。
- 子任务：
  - 提示选对逻辑并计错。
  - Web Audio 正误音效。
  - 声音开关持久化。
- DoD：
  - 提示行为符合需求定义。
  - 静音后不播放音效且状态可恢复。

#### B6. 页面完善与视觉状态
- 目标：补齐各页面完整展示。
- 子任务：
  - 首页进度展示。
  - 单元/关卡页状态展示（锁定、完成、可玩）。
  - 通关页词表与下一关逻辑。
- DoD：
  - 页面信息完整，文案与状态一致。

### C. 质量与稳定性

#### C1. 单元测试
- 覆盖目标：
  - 星级计算
  - 最佳成绩覆盖
  - 解锁规则
  - 关卡切片与补词
  - 版本迁移
- DoD：
  - 核心规则测试通过率 100%。

#### C2. E2E 测试
- 覆盖目标：
  - 端到端主链路
  - 提示计错
  - 刷新恢复
  - 非法路由回退
- DoD：
  - 主链路 E2E 全通过。

#### C3. 稳定性与性能回归
- 覆盖目标：
  - 连续通关压力（20关）
  - 页面切换响应
  - 计时器泄漏检查
- DoD：
  - 无白屏、无进度丢失、响应满足指标。

### D. 发布与验收

#### D1. 部署配置
- 子任务：
  - 配置 SPA 回退重写。
  - 构建产物校验。
- DoD：
  - 子路径访问不 404。

#### D2. 验收执行
- 子任务：
  - 按需求验收清单逐项验证。
  - 输出验收报告（通过项/风险项）。
- DoD：
  - 验收报告完整，可作为上线依据。

## 5. 优先级（MoSCoW）
- Must：A1-A3、B1-B4、C1、D1
- Should：B5、B6、C2
- Could：C3（深度性能调优）
- Won't（本期不做）：账号体系、云同步、排行榜

## 6. 依赖关系
- A1 -> A2/A3
- A3 -> B1
- B1 -> B2
- B2 -> B3/B5
- B3/B4 -> B6
- B1-B6 完成后进入 C1/C2
- C 阶段通过后进入 D 阶段

## 7. 角色建议
- 前端开发：A/B 主责
- 测试：C/D 主责
- 产品/项目：需求澄清、验收评审

## 8. 里程碑排期（建议）
- 第1周：A阶段 + B1
- 第2周：B2-B4
- 第3周：B5-B6 + C1
- 第4周：C2-C3 + D阶段

## 9. 任务验收模板
每个任务验收记录建议包含：
- 任务编号
- 完成日期
- 完成人
- 验收人
- 验收结果（通过/不通过）
- 备注与遗留问题

## 10. 风险与应对
- 需求变更频繁：冻结迭代范围，新增需求进入下一里程碑。
- 数据质量问题：建立词库校验脚本与人工抽样复核。
- 兼容性问题：优先保证主流 Chromium 端表现，其他浏览器记录兼容差异。
